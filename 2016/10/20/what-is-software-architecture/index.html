What is software architecture?

Software architecture refers to how a piece of software fits together as a whole. Large implementations of a software project are usually broken down in many different ways, and the architecture of the project will generally dictate how components are broken down.

The architecture of a project usually relies on one or more design patterns. These are reusable solutions for common problems in software engineering. Design patterns can be used throughout the life of the application, from just starting out to refactoring a 10+ year old app.

Design patterns are also applicable at varying conceptual levels. At the highest level, a particular pattern can dictate how an entire system is held together. And at the lowest levels, a developer may use a particular pattern in a single method of a small class.

For instance, a Ruby on Rails project uses Model-View-Controller (MVC) as it’s primary design pattern.

  • Models are data objects, and usually they map directly to a row in a database table. You might have a User or Contact model, with attributes like first name, last name, email address, etc.
  • Views are html representations of those models. So, you might have a profile page view where you show the data for a particular user on the screen.
  • Controllers are the piece that kick everything off. So if a request for /contacts/1234 comes in, Rails will route that to the correct controller. From there the controller will perform authorization, get the contact model from the database, and tell Rails which view to show.

There are many benefits for following a software architecture and properly using design patterns:

  • Organization of code. The architecture of a project generally dictates where you would put your code as your project grows. In the Rails example above, If you added a new concept to your app like Reservation, it would be clear where to put the code. This is highly beneficial for teams, because they don’t have to reinvent the wheel each time something new needs to be added to the project.
  • Deferring design decisions. It’s conceptually and sometimes physically expensive to commit to a particular design decision too early on in a project, especially if the requirements are in flux. You can use a pattern like template method to abstract this decision away until you have enough details to decide on a particular search technology.
  • Maintainability, extendability, and readability of the code. These are arguably the most important elements of software engineering with a team. A good software architecture and proper use of design patterns will lead developers to write code that is inherently readable and maintainable. This leads to less bugs, less churn, happier developers and happier customers.

There is also something to be said for over-engineering your architecture from the start. Joel Spolsky coined the term Architecture Astronaut - someone who over-abstracts everything, and therefore can get nothing done. And this is true! Properly architecting software and using design patterns takes skill and practice. I remember when I dove deep into learning design patterns myself, and I came away looking at everything as a pattern, which is not the right way to approach a problem. There is a proper balance for using patterns versus not, and it takes a lot of experience as a software engineer to find the right path.
